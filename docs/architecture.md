# Архитектура pjson [BR-001](../requirements/business/BR-001.json)

## Обзор [BR-001](../requirements/business/BR-001.json)

**pjson** (persistent JSON) — это C++20 header-only библиотека, предоставляющая JSON-подобный API для структур данных, хранящихся в персистентном адресном пространстве (ПАП), управляемом [PersistMemoryManager (pmm)](https://github.com/netkeep80/PersistMemoryManager).

В отличие от традиционных JSON-библиотек (например, nlohmann::json), работающих с объектами в оперативной памяти и требующих явной сериализации/десериализации, pjson работает непосредственно с персистентной памятью. Все JSON-данные хранятся в бинарном образе памяти, который можно сохранить на диск и загрузить обратно без преобразования.

## Архитектурные принципы [SR-002](../requirements/stakeholder/SR-002.json)

### 1. Единая парадигма леса AVL-деревьев [BR-002](../requirements/business/BR-002.json)

pjson использует ту же архитектуру леса AVL-деревьев, что и pmm, как для управления памятью, так и для управления хранимыми объектами. Это означает:

- **Выделение памяти** (управление свободными блоками) использует AVL-деревья
- **JSON-объекты** (карты ключ-значение) используют AVL-деревья через `pmap`
- **JSON-массивы** используют `parray` для доступа по индексу за O(1)
- **Интернирование строк** использует AVL-деревья через `pstringview`

В pjson нет отдельного пул-аллокатора или собственного управления памятью — все операции с памятью делегируются pmm.

### 2. pjson как тонкая обёртка [SR-002](../requirements/stakeholder/SR-002.json)

pjson — это *специфическая JSON-обёртка* вокруг pmm. Она добавляет:
- Семантику типов JSON-значений (null, boolean, integer, real, string, array, object)
- Адресацию по пути (`/a/b/0/c`)
- Парсинг и сериализацию JSON (кодек)
- JSON-специфичные расширения ($ref, $base64)

Она НЕ дублирует:
- Управление памятью (pmm обрабатывает выделение, освобождение, объединение блоков)
- Персистентные структуры данных (pmm предоставляет pmap, parray, pstring, pstringview, pptr)
- Бэкенды хранения (pmm предоставляет HeapStorage, MMapStorage, StaticStorage)
- Потокобезопасность (pmm предоставляет политики блокировок)

### 3. Адресация на основе смещений [NFR-002](../requirements/nonfunctional/NFR-002.json)

Все межобъектные ссылки используют индексы гранул (смещения от базы области памяти), а не сырые указатели. Это делает весь образ памяти адресно-независимым — его можно:
- Сохранить в файл и загрузить по другому базовому адресу
- Отобразить в разделяемую память
- Передать между процессами

## Модель JSON-узла [FR-002](../requirements/functional/FR-002.json)

Каждое JSON-значение представлено как узел, содержащий:
- **Тег**, идентифицирующий тип значения (null, boolean, integer, uinteger, real, string, binary, array, object, ref)
- **Payload** (union), хранящий либо:
  - Скалярные значения напрямую (boolean, integer, real)
  - Смещения (индексы гранул), указывающие на персистентные контейнеры (string, array, object, binary, ref)

Структура узла должна быть тривиально копируемой (`POD`), чтобы весь образ памяти можно было сохранить как сырые байты.

> **Примечание (FR-014):** В унифицированной архитектуре тег типа (`node_tag`) устраняется из структуры узла. Тип определяется через поле `node_type` в `TreeNode` pmm, а структура узла содержит только payload (union без дискриминатора). См. раздел [Унификация типа AVL-дерева и типа JSON-узла](#унификация-типа-avl-дерева-и-типа-json-узла-fr-014).

## Соответствие структур данных [FR-001](../requirements/functional/FR-001.json)

| Тип JSON  | Контейнер pmm | Сложность доступа | Примечания                               |
|-----------|---------------|-------------------|------------------------------------------|
| null      | (только тег)  | O(1)              | Нет payload                              |
| boolean   | (inline)      | O(1)              | Хранится напрямую в payload узла         |
| integer   | (inline)      | O(1)              | int64_t в payload                        |
| real      | (inline)      | O(1)              | double в payload                         |
| string    | pstring       | O(1) доступ       | Изменяемая, динамического размера        |
| array     | parray        | O(1) индекс       | Непрерывная, кэш-дружественная           |
| object    | pmap          | O(log n) ключ     | AVL-дерево, отсортированное по ключу     |
| binary    | parray<byte>  | O(1) индекс       | Хранение сырых байтов                    |
| ref       | pstringview   | O(1) сравнение    | Интернированная ссылка по пути           |

## Выделенные AVL-деревья для типов узлов pjson [FR-009](../requirements/functional/FR-009.json)

### Принцип [FR-009](../requirements/functional/FR-009.json)

Вместо compile-time типизации в pmm для pjson (которая по сути не нужна, поскольку узлы pjson имеют динамический тип через `node_type` в `TreeNode`), используется подход с фиксированными идентификаторами типов AVL-деревьев для каждого типа узла pjson. Каждый тип узла получает своё выделенное AVL-дерево в лесе AVL-деревьев pmm.

### Фиксированные идентификаторы типов [FR-009](../requirements/functional/FR-009.json)

| node_type | Тип узла pjson | Контейнер pmm        | Примечания                                |
|-----------|----------------|----------------------|-------------------------------------------|
| 1         | $ref           | pstringview          | Персистентные ссылки                      |
| 2         | $base64        | parray\<uint8_t\>    | Бинарные данные                           |
| 3         | pstring        | pstring              | Строковые значения                        |
| 4         | parray         | parray\<pjson_node\> | JSON-массивы                              |
| 5         | pjson          | pjson_node           | Корневые узлы JSON                        |
| (pmm)     | pstringview    | —                    | Уже имеет тип дерева в pmm               |
| (pmm)     | pmap           | —                    | Уже имеет тип дерева в pmm               |

### Преимущества подхода [FR-009](../requirements/functional/FR-009.json)

1. **Итерация по типу**: Итераторы по типизированным AVL-деревьям позволяют перебрать все узлы определённого типа (например, все массивы, все ссылки) за O(n).
2. **Индексация для СУБД**: Выделенные деревья создают естественную основу для индексов в pjson_db.
3. **Восстановление данных**: При повреждении ПАП можно обойти каждое типизированное дерево независимо и восстановить неповреждённые узлы.
4. **Полнотекстовый поиск**: Раздельные деревья для pstringview и pstring позволяют реализовать поиск по ключам и значениям независимо.
5. **Сборка мусора**: Обход дерева pstringview позволяет определить неиспользуемые интернированные строки.

## Унификация типа AVL-дерева и типа JSON-узла [FR-014](../requirements/functional/FR-014.json)

### Принцип: pptr как динамический тип [FR-014](../requirements/functional/FR-014.json)

В архитектуре с фиксированными идентификаторами типов AVL-деревьев (FR-009) тип JSON-узла хранится в двух местах:
1. **node_tag** в структуре `pjson_node` (тег типа: null, boolean, integer, ...)
2. **node_type** в `TreeNode` pmm (идентификатор AVL-дерева: 1=$ref, 2=$base64, ...)

Эта избыточность устраняется **унификацией**: тип JSON-узла определяется **исключительно** через поле `node_type` в `TreeNode` pmm. Отдельный `node_tag` в структуре узла больше не нужен.

### pptr как динамический тип [FR-014](../requirements/functional/FR-014.json)

В результате унификации `pptr` становится **динамическим типом**:
- Традиционно `pptr<T>` — это типизированный персистентный указатель, где `T` определяет тип данных на этапе компиляции
- В новой архитектуре тип данных, на которые указывает `pptr`, определяется **в runtime** через чтение поля `node_type` в `TreeNode`, к которому относится данный блок памяти
- Один и тот же `pptr` может указывать на узел любого типа pjson

```
Было (FR-009):
┌─────────────┐       ┌──────────────────────┐
│  TreeNode   │       │   pjson_node         │
│  node_type=5│──────>│   node_tag=object    │  ← дублирование типа
│  ...        │       │   payload: pmap<...> │
└─────────────┘       └──────────────────────┘

Стало (FR-014):
┌─────────────┐       ┌──────────────────────┐
│  TreeNode   │       │   payload            │
│  node_type=7│──────>│   (union без тега)   │  ← тип определяется из TreeNode
│  ...        │       │   pmap<...>          │
└─────────────┘       └──────────────────────┘
```

### Расширенная таблица идентификаторов типов [FR-014](../requirements/functional/FR-014.json)

С унификацией каждый тип JSON-узла (включая скалярные) получает свой идентификатор в лесе AVL-деревьев:

| node_type | Тип узла pjson   | Payload              | Примечания                                |
|-----------|------------------|----------------------|-------------------------------------------|
| 1         | null             | (нет payload)        | Только факт существования узла            |
| 2         | boolean          | uint32_t             | true/false                                |
| 3         | integer          | int64_t              | Знаковое целое                            |
| 4         | uinteger         | uint64_t             | Беззнаковое целое                         |
| 5         | real             | double               | Число с плавающей точкой                  |
| 6         | string           | pstring              | Изменяемая строка                         |
| 7         | object           | pmap\<pstringview,V\>| Карта ключ-значение                       |
| 8         | array            | parray\<pptr\>       | JSON-массив                               |
| 9         | binary           | parray\<uint8_t\>    | Бинарные данные ($base64)                 |
| 10        | ref              | pstringview          | Персистентная ссылка ($ref)               |
| (pmm)     | pstringview      | —                    | Уже имеет тип дерева в pmm               |
| (pmm)     | pmap             | —                    | Уже имеет тип дерева в pmm               |

### Преимущества унификации [FR-014](../requirements/functional/FR-014.json)

1. **Экономия ПАП**: Устраняется поле `node_tag` (4+ байт) в каждой структуре узла — при миллионах узлов экономия существенна.
2. **Единая точка истины**: Тип узла определяется в одном месте (`TreeNode.node_type`), что исключает рассогласование.
3. **Динамическая типизация pptr**: `pptr` может указывать на узел любого типа — тип определяется в runtime через `TreeNode`, что упрощает API и уменьшает количество шаблонных параметров.
4. **Итерация по типу**: Все преимущества FR-009 сохраняются — итерация по всем узлам определённого типа через обход соответствующего AVL-дерева.
5. **Упрощение структуры узла**: Структура узла становится чистым union payload без дискриминатора.

### Примечание о pmap\<K, V\> [FR-001](../requirements/functional/FR-001.json)

В контексте pjson ключ K в `pmap<K, V>` предположительно всегда равен `pstringview`, поскольку JSON-объекты используют строковые ключи. Это предположение требует проверки при реализации. Если подтвердится, шаблонный `pmap<K, V>` можно заменить на `pmap<pstringview, V>` для упрощения.

### Перспектива СУБД (pjson_db) [BR-003](../requirements/business/BR-003.json)

На основе pjson планируется создание:
- **pjsonRVM** — виртуальная машина (по аналогии с [jsonRVM](https://github.com/netkeep80/jsonRVM))
- **pjson_db** — персистентная СУБД

Архитектура леса AVL-деревьев с фиксированными типами узлов обеспечивает из коробки:
- Индексацию данных по типу и значению
- Полнотекстовый поиск (раздельно по pstringview и pstring)
- Разграничение прав доступа по узлам и ветвям pjson
- Восстановление данных при повреждении ПАП

## Анализ прототипа [CR-001](../requirements/constraints/CR-001.json)

### Текущее состояние (BinDiffSynchronizer) [CR-001](../requirements/constraints/CR-001.json)

Прототип в [BinDiffSynchronizer](https://github.com/netkeep80/BinDiffSynchronizer) предоставляет рабочую реализацию pjson с:
- Полной системой типов JSON (10 типов узлов)
- CRUD API на основе путей
- JSON-кодеком (парсер/сериализатор) с расширениями $ref и $base64
- Обходом дерева, поиском, клонированием, пакетными операциями
- Обширным набором тестов (25+ тестовых файлов)

### Выявленные проблемы [CR-001](../requirements/constraints/CR-001.json)

1. **Дублирование пул-аллокатора** (`pjson_pool_pmm`): Использует `PamManager::ppool<node>` для выделения узлов вместо прямого выделения через лес AVL-деревьев pmm. Это вводит отдельный слой управления памятью, дублирующий функциональность pmm.

2. **Дублирование фасада PAM** (`pam_pmm.h`): Добавляет слой Persistent Address Manager с собственным реестром именованных объектов (два экземпляра `pmap`) поверх pmm. Часть этой функциональности пересекается с тем, что предоставляет pmm.

3. **Сортированный массив-карта** (`pmap_pmm.h`): Использует сортированный массив (`parray<Entry>`) с бинарным поиском вместо `pmap` pmm на основе AVL-дерева. Это даёт O(n) для вставки/удаления против O(log n) для AVL.

4. **Глобальное состояние**: PMM использует статические переменные (паттерн мультитон), ограничивая одну БД на процесс для каждой конфигурации.

5. **Интернирование строк не освобождает память**: Словарь интернирования только растёт.

### Путь миграции [CR-001](../requirements/constraints/CR-001.json)

Миграция из BinDiffSynchronizer в этот репозиторий должна:

1. Заменить `pjson_pool_pmm` на прямое выделение через pmm
2. Заменить `pmap_pmm` (сортированный массив) на нативную `pmap` pmm (AVL-дерево)
3. Упростить или убрать фасад PAM, где он дублирует pmm
4. Сохранить модель узлов, кодек, API на основе путей и вспомогательные функции
5. Портировать набор тестов

## Формат комментариев в исходном коде [FR-005](../requirements/functional/FR-005.json)

Все исходные файлы должны ссылаться на ID требований:

```cpp
// @req FR-001 — Хранение JSON-узлов через лес AVL-деревьев pmm
class pjson_node { ... };
```

Для ссылок на несколько требований:

```cpp
/**
 * @req FR-003 — Доступ к JSON-значениям по пути
 * @req FR-004 — Парсинг и сериализация JSON
 */
```
